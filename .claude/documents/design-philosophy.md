# 設計思想

## プロジェクト概要

このドキュメントでは、本プロジェクトの設計思想と開発方針を定義します。

## 基本方針

### コード品質

- **読みやすさ優先**: 次の開発者が理解しやすいコードを書く
- **シンプルさ**: 必要最小限の実装で要件を満たす（KISS原則）
- **保守性**: 変更に強い設計を心がける

### アーキテクチャ

- **Next.js 15 App Router**: 最新のNext.js機能を活用
- **React 19**: 最新のReact機能を活用
- **TypeScript**: 型安全性を重視
- **Package by Feature**: 機能単位でコードを整理

### ディレクトリ構成（Package by Feature）

機能単位でコードを整理し、関連するコンポーネント、ロジック、スタイル、型定義を同じディレクトリに配置します。

```
src/
├── app/                    # App Router のページ
│   └── (routes)/          # ルーティング
├── features/              # 機能単位のディレクトリ
│   ├── feature-a/
│   │   ├── components/   # この機能専用のコンポーネント
│   │   ├── hooks/        # この機能専用のカスタムフック
│   │   ├── types/        # この機能専用の型定義
│   │   ├── utils/        # この機能専用のユーティリティ
│   │   └── styles/       # この機能専用のスタイル
│   └── feature-b/
│       └── ...
├── components/            # 共通コンポーネント
├── hooks/                 # 共通フック
├── types/                 # 共通型定義
├── utils/                 # 共通ユーティリティ
├── styles/                # グローバルスタイル
└── lib/                   # 外部ライブラリのラッパーや設定
```

#### Package by Feature の利点

- **高凝集性**: 関連するコードが近くに配置され、機能の全体像を把握しやすい
- **低結合性**: 機能間の依存を最小限に抑え、変更の影響範囲を限定
- **スケーラビリティ**: 機能追加時に新しいディレクトリを作成するだけで拡張可能
- **チーム開発**: 機能単位で作業を分担しやすく、コンフリクトが発生しにくい

### スタイリング

- **SCSS**: スタイル管理
- **モジュール化**: コンポーネント単位でスタイルを分離
- **クラス命名**: 独自のクラス名を使用（HTMLタグ直接指定を避ける）

### 依存関係管理

- **最小限の依存**: 本当に必要なパッケージのみ追加
- **定期的な更新**: セキュリティと最新機能のバランスを取る

## コーディング規約

### ESLint・Prettier

- シングルクォート使用
- セミコロンあり
- インデント: 2スペース

### 型定義

- `any`の使用を避ける
- 適切な型定義を作成
